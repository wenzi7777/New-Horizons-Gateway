# v0.2.8

## Fixes

### Fast offline detection for powered-off devices

- Fixed devices staying reported as connected (and therefore shown online with
  "last seen = now" in the desktop) for ~90 seconds after being powered off
  (soft shutdown or hard power-cut).
- Root cause: `GatewayState.control_stale_sec` defaulted to **90s**, so a device
  that stopped heartbeating kept being re-reported as `connected: true` /
  `findme_state: "attached"` in every gateway snapshot until the 90s window
  elapsed. This was the dominant offline-detection latency — it gated every
  downstream desktop/frontend timer.
- Lowered the default control-stale window to **16s** (≈3 missed 5s heartbeats
  plus jitter slack). A device that misses one or two heartbeats stays
  connected; a powered-off device is dropped within ~16–21s.
- Made the window operator-tunable via a new `control_stale_sec` config key
  (defaults to 16s), so it can be raised for flaky LANs without a code change.
- Defense-in-depth: a forwarded stream packet now also refreshes a device's
  liveness clock, so a heavily-streaming device never flaps offline under the
  tighter window just because a heartbeat was briefly missed. Only forwarded
  (accepted) packets count — a denied device's raw packets do not.

## Verification

- Gateway unit suite passes, including new tests for fast-detect timing,
  forwarded-vs-denied UDP liveness, and config coercion.
- Real-time end-to-end check (gateway state → desktop service) confirms a
  powered-off device flips to disconnected within the window.

## Deployment

- **This must be deployed to the running gateway to take effect** — update the
  gateway host and restart it. The desktop change alone cannot fix this; the
  stale window lives in the gateway.
